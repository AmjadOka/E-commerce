Order flow
1
POST /cart/checkout/:type
Reads cart, fetches Tax model for taxPrice+shippingPrice, builds order
2a
Cash path
Order created immediately. Stock updated only if totalOrderPrice === 0. Cart cleared.
2b
Card path
Stripe session created with line_items. Order saved with sessionId, isPaid=false.
3b
Stripe webhook fires
POST /checkout/session — validates sig, sets isPaid+isDelivered, decrements stock, clears cart, sends email.
3a
Admin confirms cash
PATCH /cart/checkout/:orderId/cash — sets paidAt, triggers updateStock + email.

The Stripe secret key and webhook secret are hardcoded in order.service.ts. Move these to environment variables (STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET) before deploying.

Stripe CLI for local testing: stripe listen --forward-to localhost:3000/api/v1/checkout/session. The endpoint requires the raw request body — ensure rawBody: true is set in your NestJS app bootstrap.

Rating recalculation runs inline after every mutation: all reviews for the product are re-fetched, summed, and averaged. Consider extracting this into a shared helper or a Mongoose post-hook to avoid the triple duplication across create / update / remove.

Request Product/api/v1/req-productnew
Allows users to request products not yet in the catalog. Users own their requests — they can only read, edit, and delete their own. Admins have full read access. Ownership is enforced at the service level by comparing req.user._id to the stored user ObjectId.
MethodRouteDescriptionAccess
POST/Create request. 400 if user already requested same titleNeed. Body: titleNeed, details, qauntity, category?.User
GET/All requests. Populated user (no password/role). Returns length.Admin
GET/:idSingle request. User can only view own. Admin can view any.Admin · User
PATCH/:idUpdate own request only. Ownership verified in service.User
DELETE/:idDelete own request only. Returns void.User

In findOne (service), the ownership check runs before the null check — if the document doesn't exist, requestProduct?.user._id will throw rather than returning 404. Swap the order: check for null first, then authorize.

Tax/api/v1/texnew
Singleton tax and shipping configuration. Only one record exists in the collection at any time. POST acts as upsert — it creates the record if none exists, or updates the existing one. All three routes are admin-only. The taxPrice and shippingPrice values are consumed by the Order module when calculating totalOrderPrice.
MethodRouteDescriptionAccessBody

Image Upload/api/v1/imagenew
Cloudinary-backed file upload via multipart/form-data. Two endpoints handle single and bulk uploads. Files are streamed directly to Cloudinary using Node.js Readable.from(buffer) — no temp files on disk. The module exports both the service and provider so other modules can inject CloudinaryService directly.
UploadFilesController /api/v1/image
MethodRouteDescriptionAccess
POST /upload Upload a single file. Field name: file. Max size: 5 MB. Allowed types: jpg, jpeg, png, webp. Returns the full Cloudinary UploadApiResponse. Admin · User
POST /uploads Upload up to 5 files. Field name: files[]. Max size per file: 500 KB. Allowed type: image/png only. Returns an array of Cloudinary responses. Admin
Request format
Single upload — /upload
Content-Type: multipart/form-data
fileFilereq · max 5 MB · jpg/jpeg/png/webp
Multi upload — /uploads
Content-Type: multipart/form-data
files[]File[]req · max 5 files · 500 KB each · png only

Cloudinary credentials (cloud_name, api_key, api_secret) are hardcoded in cloudinary.provider.ts. Move these to environment variables (CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET) before deploying.

The UploadFilesModule exports both CloudinaryService and CloudinaryProvider. Import this module in any feature module (e.g. Product) that needs to attach images during create/update.
